# 07. Когда заблокировали: план действий

Рано или поздно конкретный IP могут заблокировать. Это **штатная ситуация**, а не
поломка. Признаки: клиент пишет «подключено», но интернета нет; сервер перестал
пинговаться из РФ.

## Дерево решений

```
Перестал работать VPN
│
├─ Сервер пингуется/открывается из РФ?   (./scripts/check-ip-russia.sh)
│     ├─ НЕТ → IP заблокирован
│     │         → сменить IP / перейти на второй сервер
│     │         → или вход через CDN (схема B) / CFXHTTP (схема C)
│     │
│     └─ ДА → IP жив, режут протокол/соединение
│               ├─ Reality «долго коннектится / рвётся»?
│               │     → перейти на VLESS + XHTTP (схема A)
│               │     → сменить SNI/донора
│               ├─ Это мобильный оператор?
│               │     → режим белых списков: SNI из разрешённых,
│               │       Hysteria2 + port hopping, multi-hop через РФ-облако
│               └─ Нужен только конкретный сайт, не туннель?
│                     → клиентский обходчик DPI (zapret/GoodbyeDPI)
│
└─ Заблокировано вообще всё (и серверы, и CDN)?
           → Tor + мосты (snowflake/webtunnel)
           → достать оффлайн-чемоданчик, поднять новый сервер с нуля
```

## Порядок реанимации (от простого к сложному)

1. **Убедитесь, что это блокировка IP, а не ваша ошибка.** Зайдите по SSH (если
   SSH идёт с другого канала, но не из РФ — IP, скорее всего, блокирован для РФ).
   Проверьте `sudo ./deploy.sh health` и логи (см. [09-debug.md](09-debug.md)).
2. **Смените SNI/донора.** Иногда режут по связке/поведению — попробуйте другого
   донора (правьте `REALITY_SNI`/`REALITY_DEST` в `.env`, затем `sudo ./deploy.sh reality`).
3. **Смените порт** (но 443 — самый незаметный; меняйте только при основаниях).
4. **Смените IP** (самое надёжное):
   - закажите новый IP/VPS,
   - восстановите из бэкапа: `sudo ./deploy.sh all` (ключи переиспользуются),
   - перевыпустите ссылки. См. [05-backup-restore.md](05-backup-restore.md).
5. **Переключитесь на резервную схему** через клиент с автопереключением
   (Hysteria2 / CDN / CFXHTTP) — в идеале это происходит само (url-test).
6. **Mobile-only проблема?** → multi-hop через РФ-облако (см.
   [08-defense-in-depth.md](08-defense-in-depth.md)).

## Мобильные «белые списки»

К 2026 мобильные операторы РФ (МТС, Мегафон, Tele2, Билайн, Yota) местами перешли
на **белые списки**: пропускается трафик только к одобренным IP, остальное
режется. Даже идеальный Reality на заграничном IP не пройдёт, потому что сам
заграничный IP не в белом списке.

**Решение (multi-hop):** промежуточное звено внутри РФ на IP, который уже в белом
списке оператора (дешёвая виртуалка в российском облаке), и через него — на
заграничный сервер:

```
Телефон (мобильный оператор)
   → промежуточный VPS в РФ-облаке (IP в белом списке)
        → ваш заграничный VPS (VLESS+Reality)
           → свободный интернет
```

Сложнее и дороже (две машины) — беритесь, только если на мобильном обычная схема
не работает. На домашнем проводном интернете обычно достаточно базовой схемы.

## Профилактика

- Держите **2 VPS** у разных хостингов в разных странах.
- Настройте **мониторинг** доступности IP из РФ (`check-ip-russia.sh` в cron) и
  Telegram-алерты (`healthcheck.sh`) — узнавайте о блокировке раньше пользователей.
- Обновляйте Xray-core — свежие версии лучше обходят новые эвристики.
